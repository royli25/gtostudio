# WASM Multithreading Debug Log

## Status
Threading is technically working but only providing ~25% speedup with 10 threads (expected 5-10x). The global rayon pool has 10 threads, sub-workers are alive and running, but `solve_step` only dropped from ~1600ms to ~1270ms.

## Architecture

### How it works
1. **Main solver-worker** (`web/public/solver-worker.js`): A Web Worker that loads the WASM module and runs the solve loop
2. **Sub-workers** (spawned by `web/public/snippets/solver-wasm-18e4a20c677cf2b4/workerHelpers.js`): 10 Web Workers that share the same `SharedArrayBuffer` memory and run rayon's work-stealing loop
3. **Rayon global pool**: Built via `build_global()` with a custom `spawn_handler` that sends `ThreadBuilder`s through an `spmc` channel to the sub-workers
4. **Engine parallelism**: The engine's `for_each_child()` in `engine/src/utility.rs:16` calls `node.action_indices().into_par_iter().for_each(op)` when `node.enable_parallelization()` returns true (i.e., `self.river == NOT_DEALT`)

### Key files
- `engine/wasm/src/lib.rs` — WASM bindings, GameManager struct
- `engine/wasm/src/rayon_adapter.rs` — Rayon-to-WebWorker bridge, thread pool setup
- `engine/wasm/workerHelpers.js` — Sub-worker lifecycle (generated by wasm-bindgen, patched in web/public/)
- `engine/src/utility.rs:16` — `for_each_child()` parallel dispatch
- `engine/src/solver.rs:136` — `solve_recursive()` uses `for_each_child()` at lines 181, 228, 340
- `engine/src/game/node.rs:202` — `enable_parallelization()` returns `self.river == NOT_DEALT`
- `engine/src/mutex_like.rs` — `MutexLike<T>` is NOT a real mutex, just `UnsafeCell` for interior mutability

## What we verified

### Working
- `crossOriginIsolated: true` in the worker context
- WASM memory is SharedArrayBuffer (flags=0x03, imported, shared)
- 10 sub-workers load WASM, init with shared memory, and enter rayon worker loop
- `rayon::current_num_threads()` returns 10 in the global pool
- Simple parallel sum `(0..10M).into_par_iter().map(|x| x % 7).sum()` returns correct result (29999994) — proves rayon distributes work to sub-workers
- `workerHelpers.js` sub-workers use new object-based API: `pkg.default({ module_or_path: data.module, memory: data.memory, thread_stack_size: 1048576 })`

### The problem
- `solve_step` takes ~1270ms (was ~1600ms with custom pool + `pool.install()`, and ~1600ms single-threaded with non-shared WASM)
- `exploitability` takes ~780ms (was ~1000ms)
- Net: only ~25% speedup from 10 threads, suggesting most work is still single-threaded

## 2026-05-22 investigation note

Two concrete build/runtime issues were found:

1. The WASM crate enabled the engine's `rayon` feature but not `custom-alloc`.
   `engine/src/alloc.rs` explicitly calls out the default allocator as inefficient in multithreaded WASM, and the recursive solver allocates temporary `Vec`s at every node. This can erase most of the benefit from Rayon workers. `engine/wasm/Cargo.toml` now enables `postflop-solver/custom-alloc` for the WASM build.
2. `engine/wasm/workerHelpers.js` still used the old positional `wasm-bindgen` init API, while the generated/public snippet had been manually patched to the object-based API with `module_or_path`, shared `memory`, and `thread_stack_size`. Future rebuilds could regenerate a broken snippet from the stale source. The source helper now matches the working generated/public version.

After rebuilding the WASM package, re-run the benchmark table below before changing solver parallelism. If speedup is still flat, the next most likely bottleneck is limited parallel work below river chance nodes.

## What we tried

### 1. Custom thread pool with `pool.install()` (DIDN'T HELP)
- Created a `rayon::ThreadPool` stored in `static mut THREAD_POOL`
- Called `pool.install(|| solve_step(&self.game, iteration))`
- `pool.install()` sets the pool in thread-local storage for the closure
- Result: zero speedup. `solve_step` timing identical to single-threaded
- Theory: `pool.install()` may not propagate through WASM TLS correctly to the engine crate's `into_par_iter()` calls

### 2. Global pool with `build_global()` (PARTIAL IMPROVEMENT)
- Changed to `rayon::ThreadPoolBuilder::new().num_threads(N).spawn_handler(...).build_global()`
- Engine's `into_par_iter()` uses global pool by default
- `rayon::current_num_threads()` now returns 10
- Result: ~25% speedup. Better than pool.install() but far from linear

## Hypotheses for limited speedup

### H1: Atomic memory overhead (LIKELY)
Compiling with `+atomics` and `--shared-memory` makes ALL memory accesses use atomic instructions, even single-threaded code paths. This overhead (~2-3x on ARM) may negate most parallelism gains. The non-shared WASM binary (without atomics) ran at similar speed to the 10-thread shared version.

Related: before 2026-05-22, the WASM crate did not enable `custom-alloc`, so the shared-memory build was also paying default allocator overhead in recursive solver hot paths.

### H2: Limited parallel work granularity
`enable_parallelization()` only returns true when `self.river == NOT_DEALT`. The parallel dispatch only happens at:
- Flop betting nodes (2-3 actions typically)
- Turn chance nodes (~47 cards)
- Turn betting nodes (2-3 actions)
- River chance nodes (~46 cards) — but these are at `river == NOT_DEALT` since the river is *being* dealt

River betting nodes (deepest, most numerous) are sequential. If most CPU time is spent in river subtrees, parallelism only helps at the coarse level.

### H3: Rayon work-stealing overhead in WASM
Rayon's work-stealing uses futex (`Atomics.wait()`/`Atomics.notify()`) for thread parking. The overhead of cross-worker synchronization via SharedArrayBuffer may be higher than native futex.

### H4: Memory bandwidth saturation
The 1.3GB game tree is large. If the solve is memory-bandwidth-bound (random access pattern across a large tree), adding threads won't help because M1's memory bandwidth is shared across all cores.

### H5: `for_each_child` tasks are too coarse or unbalanced
When parallelizing over 2 actions at the root, one subtree might be 10x larger than the other. Rayon's work-stealing should handle this, but if the large subtree runs sequentially internally (all river nodes), one thread gets all the deep work.

## Recommended next steps

1. **Profile where time is actually spent**: Add atomic counters to `for_each_child` to measure how often the parallel vs sequential path is taken, and how many items are in each parallel dispatch.

2. **Compare against non-shared baseline**: Build a non-shared-memory WASM (without `--shared-memory` and atomics) and measure single-threaded performance. If non-shared is 2x faster, then atomic overhead is the dominant factor.

3. **Try fewer threads**: Test with 2-4 threads instead of 10. If 2 threads gives ~20% and 10 threads gives ~25%, the problem is work distribution, not overhead.

4. **Check if rayon actually runs tasks on sub-workers**: Use `std::thread::current().id()` or an atomic counter inside the `for_each_child` closure to verify multiple threads execute the parallel tasks.

5. **Look at the original wasm-postflop app**: The b-inary/wasm-postflop Vue.js app presumably achieves good multi-threaded performance. Compare their build process and runtime setup. Their repo: https://github.com/b-inary/wasm-postflop

6. **Consider wasm-bindgen-rayon crate**: Instead of the manual rayon adapter, try the official `wasm-bindgen-rayon` crate which may handle TLS and pool setup more correctly.
